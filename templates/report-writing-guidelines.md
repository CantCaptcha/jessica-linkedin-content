# Report Writing Guidelines

**Persona:** Seasoned Senior Analyst, Technology Company
**Purpose:** Professional, data-driven, business-focused reporting

---

## Core Principles

### What + Why Rule
Every assertion must be backed by rationale.

- **What we recommend** → **Why it's the right choice**
- **What the data shows** → **Why it matters**
- **What's next** → **Why this sequence makes sense**

No dangling claims. Everything requires justification.

---

## Writing Style

### Do
- Get straight to the point
- Use specific, meaningful language
- Every word earns its place
- Be concise but complete
- Write with confidence, not arrogance
- Focus on results and impact
- Back statements with evidence

### Don't
- Use cliched business phrases:
  - "In today's fast-paced world"
  - "At the end of the day"
  - "Game-changer"
  - "Synergy"
  - "Moving the needle"
  - "Paradigm shift"
- Include empty phrases that don't add information
- Use filler words
- Be verbose for the sake of sounding formal

---

## Layout & Formatting

### Visual Principles
- **White space is your friend** — give the reader room to breathe
- **Consistent formatting** — same style for same type of content
- **No walls of text** — break it up with structure
- **Scannable** — main points visible on a skim
- **Logical flow** — sections lead naturally into each other

### Structure Tools

**Headings and Subheadings**
- Clear hierarchy for organization
- Use H1 for title, H2 for major sections, H3 for subsections

**Bullet Lists**
- For related items that don't need order
- Keep items parallel in structure
- Limit to 3-7 items when possible

**Numbered Lists**
- For sequences or steps
- For prioritized items
- For processes with specific order

**Paragraphs**
- Short: 2-3 sentences maximum
- One idea per paragraph
- Start with the most important point

**Emphasis**
- Use bold sparingly for key terms or critical information
- Use italics for terms or subtle emphasis
- Don't overdo it — loses impact

**Separators**
- Use horizontal rules between major sections
- Use extra line spacing before new topics
- Break dense content into digestible chunks

## Report Structure

### Standard Sections

**1. Title Block**
- Report title (clear, specific)
- Date
- Prepared for / Prepared by
- Task or context (if applicable)
- Due date (if applicable)

**2. Executive Summary**
- Brief overview (2-3 paragraphs max)
- Key recommendation
- Critical reasoning
- Bottom line decision point

**3. Background**
- Context for the report
- Why this matters now
- What prompted this work

**4. Analysis / Evaluation**
- What was considered
- How data was collected or assessed
- Key findings
- Evidence for each conclusion

**5. Recommendation**
- Primary recommendation (what)
- Rationale (why)
- Alternatives considered and rejected (why not)

**6. Implementation Plan**
- Clear phases or steps
- Timeline
- Resources needed
- Success criteria

**7. Expected Benefits / Risks**
- What success looks like
- Potential downsides
- How risks will be mitigated

**8. Next Steps**
- Actionable checklist
- Owner and timeline for each
- Dependencies

**9. Conclusion**
- Final recommendation
- Call to action
- Contact for questions

## Formatting Standards

### Headings
```markdown
# Report Title (H1)
## Major Section (H2)
### Subsection (H3)
```

### Emphasis
- **Bold**: Key terms, critical data, decisions
- *Italics*: Technical terms, subtle emphasis

### Lists
- Bullet: Related items, options, criteria
- Numbered: Steps, sequences, priorities

### Callouts
> Use blockquotes for important notes or warnings

### Code / Technical
Use code blocks for technical specifications, commands, or data

---

## Quality Checklist

Before submitting any report, verify:

- [ ] Every "what" has a corresponding "why"
- [ ] No cliched phrases or empty business jargon
- [ ] Layout includes adequate white space
- [ ] No unbroken text walls (>4 sentences)
- [ ] Headings create clear hierarchy
- [ ] Formatting is consistent throughout
- [ ] Recommendations are actionable
- [ ] Evidence supports all claims
- [ ] Executive summary captures key points
- [ ] Next steps are clear and assigned

---

## Examples

### Good: Specific and Clear
> **Recommendation:** Adopt Claude Max 5 for the development team.
>
> **Why:** Strong coding capabilities, proven performance in technical environments, and optimal cost-per-seat at $100/month per user. The team's existing open source workflow requires no additional tool purchases.

### Bad: Vague and Cliched
> In today's fast-paced world, we need to leverage AI tools to stay ahead. We recommend Claude because it's a game-changer that will help us work smarter, not harder.

---

**Remember:** Good analysis is clear, concise, and complete. The reader should understand what you're recommending, why you're recommending it, and what happens next — all without wading through fluff.
